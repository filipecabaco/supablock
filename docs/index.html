superblock mirrors the Supabase Management API as a browsable directory tree. Log in once with a personal access token, mount, and inspect organizations, projects, config, keys and functions with the tools you already trust — ls, cat, grep, diff.

superblock — zsh

Why a filesystem

Everything is a file. Now your infrastructure is too.

No SDK, no dashboard tab farm, no one-off scripts. Your whole account becomes greppable, diffable and scriptable, and the guarantees are built in:

mount -o ro

Physically read-only. Only GET requests ever leave the box, and the kernel itself rejects every write with EROFS.

ttl + single-flight

Cached & rate-limit friendly. ls -R costs one request per endpoint, not hundreds. A 429 degrades to EAGAIN — never a hang.

sbp_…f23a

Token-safe. Stored at mode 0600, never logged, never rendered into the tree. Secret API keys stay redacted unless you opt in.

Install

One line — or build from source

The quickest path is the installer: it downloads the single-file binary for your platform from the GitHub releases (built by CI, no Erlang/Elixir needed). On Linux the FUSE userspace library is built in; on macOS install macFUSE or FUSE-T first. 

$ curl -fsSL https://filipecabaco.github.io/supablock/install.sh | sh

Building from source instead? mise pins the whole toolchain — mise install gives you Erlang, Elixir and Zig. You also need a C compiler and FUSE — on Linux that's libfuse3-dev, fuse3 and pkg-config. 

# Debian / Ubuntu prerequisites
$ sudo apt install libfuse3-dev fuse3 pkg-config

# build the release
$ git clone https://github.com/filipecabaco/supablock.git && cd supablock
$ mix deps.get
$ MIX_ENV=prod mix release

# put the launcher on your PATH
$ ln -sf "$PWD/bin/superblock" ~/.local/bin/superblock

Then let the tool check its own environment — every failed check comes with a fix hint: 

$ superblock doctor
✓ /dev/fuse present
✓ unmount tool on PATH (fusermount3)
✓ config dir mode 0700
✓ credentials mode 0600
✓ efuse port binary compiled

Quick start

Three commands to a mounted account

1

Authenticate

Create a personal access token at app.supabase.com → Account → Access Tokens, then paste it at the hidden prompt. It's validated against the API before anything is written, and stored at mode 0600.

$ superblock login
Paste your Supabase personal access token (sbp_...):
✓ Token valid — authenticated, 2 organizations found.
  Stored in ~/.config/superblock/credentials (mode 0600).
2

Pick a mountpoint

Set it once in config, or pass it to mount each time — the command-line argument always wins.

$ superblock config set mountpoint /mnt/supabase
3

Mount and browse

mount runs in the foreground; Ctrl-C unmounts cleanly (so do SIGTERM and even kill -9 — a watchdog in the FUSE port never leaves a stale mount). Explore from any other shell.

$ superblock mount
Mounted at /mnt/supabase. Ctrl-C to unmount.

One-liners

Your account, greppable

Output is deterministic — JSON is pretty-printed with sorted keys — so byte sizes are exact and diff between projects is clean.

Which projects disable signups?

$ grep -l '"disable_signup": true' /mnt/supabase/organizations/*/projects/*/config/auth.json

Compare two projects' auth config.

$ diff /mnt/supabase/organizations/my-org/projects/{refA,refB}/config/auth.json

Health of every service, everywhere.

$ cat /mnt/supabase/organizations/*/projects/*/health

All site URLs across the account.

$ grep -r '"site_url"' /mnt/supabase/organizations/*/projects/*/config/auth.json

Security

Read-only, by construction

The write path does not exist

Every request superblock can make is a GET — there is no code path for anything else. The FUSE layer exposes no mutating callbacks, and the mount itself is created with -o ro, so touch, rm, mkdir and every other write fails in the kernel with EROFS before it ever reaches userspace.

  • The token lives in ~/.config/superblock/credentials (mode 0600, dir 0700), is never logged, and shows up masked (sbp_…f23a) in status.
  • api-keys/secret renders as REDACTED — run: superblock config set expose_secrets true until you explicitly opt in. The publishable key is always visible.
  • SUPERBLOCK_TOKEN overrides the stored credential — the only environment variable in play, intended for CI.
  • Request deadlines (http_timeout_ms, default 8000) turn slow calls into EIO; a rate-limit turns into EAGAIN. The filesystem never hangs.

Troubleshooting

When something's off

"Transport endpoint is not connected"
A stale mount from a crashed session. Run superblock mount again — it recovers stale mounts automatically — or fusermount3 -u <mountpoint>. If the VM is killed outright, the port process notices and unmounts by itself.
Reads fail with "Resource temporarily unavailable"
That's EAGAIN: the Management API rate-limited you. Wait a moment and retry; superblock status shows the last-seen remaining budget per scope.
Mount fails outright
Run superblock doctor. It checks /dev/fuse, unmount tools, file permissions and the compiled FUSE port, and prints a fix per failed check (exit code 4 if anything fails).
Where are the logs?
~/.local/state/superblock/superblock.log while mounted; pass --verbose to mount for debug level. Tokens are scrubbed from all log output.
